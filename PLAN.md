# RIFLE.GG — 개발 계획

로블록스 [RIVALS](https://www.roblox.com/games/17625359962/RIVALS)에서 영감을 받은 웹 기반 1인칭 슈터.
브라우저에서 바로 플레이 가능하며, 봇 대전(오프라인)과 온라인 1v1 멀티플레이를 모두 지원하는 것이 목표.

## 확정된 방향

| 항목 | 결정 |
|---|---|
| 게임 이름 | **RIFLE.GG** |
| 플레이 방식 | 봇 대전 우선 구현 → 온라인 멀티(방 코드 기반 1v1) 추가 |
| 첫 게임 모드 | 1v1 듀얼 (라운드제, 5라운드 선취 승리) |
| 그래픽 | Three.js 기반 3D 로우폴리(로블록스풍 블록 스타일), 1인칭 시점 |
| 무기 규모 | 슬롯당 2~3종, 총 8~10종 (주무기/보조무기/근접/유틸리티 4슬롯) |

## 기술 스택

- **클라이언트**: TypeScript + Vite + Three.js
  - 물리/충돌: 자체 AABB 충돌 + 고정 타임스텝(120Hz) — FPS에 필요한 만큼만, 물리엔진 미사용
  - 상태 관리: 게임 루프 중심의 단순 구조 (프레임워크 없음)
- **서버 (Phase 5)**: Node.js + TypeScript + WebSocket (Colyseus 또는 ws 직접 사용)
  - 서버 권위(server-authoritative) 위치 검증 + 클라이언트 예측/보간
- **배포**:
  - 클라이언트: GitHub Pages (봇 모드는 서버 없이 동작)
  - 게임 서버: Render / Fly.io 무료 티어 (멀티 단계에서 결정)

## 개발 단계

### ✅ Phase 0 — 프로젝트 셋업 (완료)
- Vite + TypeScript + Three.js 초기화, 기본 씬(맵, 조명, 안개)
- Pointer Lock(마우스 잠금) 및 렌더 루프

### ✅ Phase 1 — FPS 코어 (완료)
- 1인칭 컨트롤러: WASD, 점프(코요테 타임), 대시(쿨다운 2초), 슬라이드
- 고정 타임스텝 물리(120Hz) — 프레임레이트와 무관하게 동일한 게임 속도
- AABB 충돌 + 계단 스텝업(0.55m), 로우폴리 대칭 1v1 맵 "Foundry"

**Phase 1 추가 고려사항 (검토에서 확인된 항목)**
- [x] 저사양 기기에서 물리 슬로모션 → 고정 타임스텝으로 해결
- [x] Esc 후 재입장 시 pointer lock 쿨다운 예외 처리
- [x] 고주사율(144/240Hz) 모니터에서 점프/대시 입력 드랍 → 물리 스텝이 소비한 뒤에만 pressed 클리어
- [x] 대시 중 점프 씹힘·접지 판정 소실 → 대시가 수직 속도를 덮어쓰지 않도록 수정(감소 중력 적용)
- [x] 공중에서 스텝업으로 중앙 벽(3.6m) 맨틀 가능하던 익스플로잇 → 직전 스텝 접지 상태로 게이트
- [x] Ctrl 슬라이드 바인드가 Ctrl+W(탭 닫기)와 충돌 → C 키만 사용, 게임 키 preventDefault
- [x] 낙하 중 카메라가 머리 위로 떠오르는 이징 누출 → 스탠스 눈높이 오프셋만 이징
- [x] 모바일: 가로모드 강제(전체화면+orientation lock) + 가상 조이스틱/시점 드래그/터치 버튼
- [ ] 고속(대시 22m/s) + 얇은 콜라이더 조합의 터널링 여부 지속 확인 (120Hz 스텝당 최대 0.18m 이동이라 현재 맵(최소 두께 0.9m)에서는 안전)
- [ ] 감도/FOV 설정은 Phase 4에서 노출

### ✅ Phase 2 — 전투 시스템 (완료)
- 무기 프레임워크: 히트스캔(레이캐스트) + 투사체(수류탄) + 근접(칼)
- 무기 7종: 돌격소총·샷건·저격총 / 권총·우지 / 칼 / 수류탄 (같은 슬롯 키 재입력으로 교체)
- 데미지/HP(헤드샷 배율), 히트마커, 킬 피드, 연습용 표적 더미 6기(머리/몸 히트박스, 리스폰)
- HUD: 동적 크로스헤어, 탄약/장전, 체력바(전투 외 리젠), 슬롯 표시
- 뷰모델(절차 생성 박스 모델) + 반동(뷰 펀치)/블룸 탄퍼짐, ADS 줌(FOV+감도)

**Phase 2 추가 고려사항 (반영 현황)**
- [x] 물리 월드 레이캐스트 API (slab test, 히트 대상 식별 포함)
- [x] 머리/몸 히트박스 분리 — `Damageable`/`Hitbox` 인터페이스로 봇·원격 플레이어 공용
- [x] 무기 데이터 테이블(`weapons.ts`) 분리
- [x] 탄퍼짐 시드 기반 결정적 RNG(mulberry32) — Phase 5 서버 재현 대비
- [ ] 무기별 사운드/이펙트 고도화는 Phase 4
- [ ] 라운드 스코어 HUD는 Phase 3에서 게임 루프와 함께

### ✅ Phase 3 — 1v1 듀얼 게임 루프 (봇 대전) (완료)
- 라운드 구조: 카운트다운(3초) → 전투 → 킬 시 라운드 종료 → 5라운드 선취 승리 → 결과
- 메뉴에서 모드(연습장/1v1 대전)·난이도·로드아웃(주무기/보조무기) 선택
- 봇 AI: 중앙 통로 웨이포인트 라우팅 + 끼임 감지(점프/사이드스텝), 교전 시 스트레이프, 난이도별 조준 오차·반응 속도·버스트 길이
- 스코어 HUD, 라운드 배너, 봇 체력바; 듀얼 중 연습 표적 비활성화

**Phase 3 추가 고려사항 (반영 현황)**
- [x] 봇이 플레이어와 동일한 `PlayerController` 물리를 카메라 없이 사용 (`ControlSource` 인터페이스로 결합 제거)
- [x] 게임 상태 머신(`DuelManager`) 분리 — Phase 5에서 서버가 동일 상태 머신 소유 가능
- [x] 봇 조준 오차·반응 속도·버스트를 난이도 파라미터로 노출 (쉬움/보통/어려움)
- [ ] 봇 무기 다양화(현재 AR 고정)와 수류탄 사용은 확장 항목

### ✅ Phase 4 — 폴리시 (완료)
- 사운드: WebAudio 절차 합성(에셋 0개) — 무기별 발사음, 장전/교체, 히트마커/킬, 피격, 발소리/점프/착지/대시/슬라이드, 폭발, 카운트다운/라운드/승패 징글, 봇 사격음(거리 감쇠)
- 총구 화염(포인트 라이트 + 글로우), 피격 시 붉은 비네트
- 설정 메뉴: 마우스 감도(0.3~3x), 볼륨, FOV(60~110) — localStorage 저장
- 남은 폴리시(선택): 사망 연출 고도화, 터치 버튼 배치 커스텀, 폭발음 거리 감쇠

### ✅ Phase 5 — 온라인 멀티플레이 (1v1) (완료)
- Node.js WebSocket 게임 서버(`server/`): 4자리 방 코드 생성/입장, 준비 핸드셰이크
- 서버 권위 상태: HP·스코어·라운드 상태머신(5선승)은 서버가 소유
- 이동은 클라이언트 시뮬레이션 + 20Hz 스냅샷 릴레이, 상대는 100ms 지연 보간 렌더링
- 사격은 슈터 판정 → 서버가 무기별 데미지 상한 + 초당 클레임 수 제한으로 검증
- 수류탄은 시각 릴레이(데미지는 던진 쪽 클레임), 연결 끊김/방 유휴(10분) 정리, ping/pong 생존 확인

**Phase 5 추가 고려사항 (반영 현황)**
- [x] 연결 끊김·상대 이탈·방 유휴 타임아웃 처리
- [x] 기초 치팅 방지(데미지 상한, 레이트 리밋) — 위치 검증·서버 재시뮬레이션은 향후 강화 항목
- [ ] 물리/전투 코드 `shared/` 분리 + 틱 입력 직렬화 → 완전한 서버 권위 시뮬레이션 (확장 단계)
- [ ] 무료 티어 콜드스타트: 첫 접속 시 서버 웨이크업 ~30초 → 연결 중 안내 표시로 대응

### ✅ Phase 6a — 팀전 M1: 로컬 팀 배틀 (완료)
- 1v1~4v4 봇 팀전: 나+아군 봇 vs 적 봇, 팀 전멸 시 라운드 종료(5선승)
- 봇 동적 타겟 선택(최근접 생존 적), 봇 대 봇 교전, 아군 오사 방지(총알/근접/수류탄)
- 팀별 스폰 4개, 생존 핍 HUD, 사망 시 관전 대기
- 남은 팀전 단계: M2 온라인 로비(최대 8인, 빈 슬롯 방장 시뮬레이션 봇), M3 폴리시(이름표/팀 색상 표시/스코어보드/맵 확장)

### Phase 6 — 확장 (선택)
- 추가 모드(FFA, 건 게임), 추가 무기/맵, 해금·상점, 코스메틱

## 알려진 이슈 / 백로그

- 헤드리스/저사양 환경에서 12fps 미만이면 게임이 의도적으로 슬로모션 (스파이럴 방지 상한 10스텝/프레임)
- 정적 그림자 캐시(`shadowMap.autoUpdate=false`) — Phase 2에서 움직이는 캐스터(플레이어 모델, 투사체) 추가 시 재활성화 필요
- 아키텍처 백로그(코드 리뷰 altitude 앵글에서 식별, Phase 2~5에서 해소):
  - 컨트롤러-카메라 결합 분리 (봇/서버 헤드리스 시뮬레이션용)
  - PhysicsWorld에 레이캐스트/스윕 쿼리 + 히트 대상 식별 추가 (Phase 2 히트스캔 전제)
  - 입력을 틱 단위 커맨드로 직렬화 (Phase 5 예측/리컨실리에이션 전제)
  - 맵을 데이터 기반 기술(박스 스펙 배열)로 분리해 서버가 메시 없이 콜라이더만 빌드
  - 디버그 핸들을 엔티티 목록 기반으로 확장 (봇 추가 시 스모크 테스트용)
